# 导演台 UI/UX 规格文档 V1

- 文档日期：2026-04-17
- 适用范围：导演台前端三页主流程（首页 / 任务历史页 / 任务详情页）
- 目标：在不改变后端能力边界的前提下，提升任务创建、追踪与失败重跑的可读性、可控性与可恢复性
- 依据：`docs/plans/2026-04-17-director-console-ui-ux-optimization-implementation-plan.md`

## 1. 产品目标与用户路径

### 1.1 产品目标

导演台的核心目标不是“展示复杂流水线”，而是让使用者用最少的认知成本完成三件事：

1. 创建任务：快速提交 Episode ID、源视频路径和工作目录，明确知道会发生什么。
2. 追踪任务：在任务历史与详情页中，快速判断当前处于哪个阶段、是否卡住、最终产物是否已生成。
3. 失败重跑：在失败后能够明确选择重跑起点，避免重复执行已经成功的阶段。

### 1.2 核心用户路径

#### 路径 A：创建任务

1. 用户进入首页。
2. 先看到产品定位、表单入口和任务历史快捷入口。
3. 填写 `Episode ID`、`源视频路径`、`工作目录`。
4. 提交后系统进入加载状态，按钮与表单进入不可重复提交状态。
5. 创建成功后自动跳转到任务详情页，并保留 `episode_id` 作为历史页联动条件。

#### 路径 B：追踪任务

1. 用户从首页快捷入口进入任务历史页，或从详情页返回历史页。
2. 先按当前 Episode 查看该 Episode 下最近任务列表。
3. 在列表中快速识别任务状态、失败阶段和更新时间。
4. 进入详情页后查看总体状态、阶段状态、失败原因与最终产物。

#### 路径 C：失败重跑

1. 用户在详情页发现任务失败。
2. 页面直接暴露“重跑”入口，并默认把失败阶段作为重跑起点候选值。
3. 用户可调整 `start_stage`，避免从头重跑。
4. 提交后进入重跑中状态，成功后自动跳转到新任务详情页。
5. 如果重跑仍失败，页面保留失败原因与历史痕迹，方便再次决策。

## 2. 信息架构

### 2.1 首页

首页承担“创建任务”的主责任，同时提供“去任务历史”的次入口。

- 核心模块
  - 产品标题与一句话说明
  - 任务创建表单
  - 任务历史快捷入口
  - 表单错误提示区

- 首页必须一眼看到的信息
  - 这是一个可直接提交任务的入口
  - 当前输入将影响一个新的任务执行
  - 提交成功后会跳转到详情页
  - 可随时查看该 Episode 的任务历史

### 2.2 任务历史页

任务历史页承担“追踪任务”的主责任。

- 核心模块
  - 当前 Episode 标识
  - 最近任务列表
  - 每条任务的状态、失败阶段、更新时间与详情入口
  - 空状态引导

- 列表信息优先级
  1. 任务状态
  2. 失败阶段或异常线索
  3. 更新时间
  4. 详情入口

- 信息密度原则
  - 以“扫一眼就能判断是否值得点开”为目标
  - 不在列表页展开完整阶段表，避免信息噪声

### 2.3 任务详情页

任务详情页承担“深度追踪与失败重跑”的主责任。

- 核心模块
  - 总体状态卡
  - 阶段状态列表
  - 失败原因卡
  - 最终产物卡
  - 重跑表单（仅失败态展示）
  - 返回历史页入口

- 详情页信息顺序
  1. 总体状态
  2. 失败原因或当前运行提示
  3. 阶段状态
  4. 最终产物
  5. 重跑操作

### 2.4 状态矩阵（页面呈现规范）

| 状态 | 首页 | 任务历史页 | 任务详情页 |
|---|---|---|---|
| `pending` | 创建成功后跳转详情，不额外展示卡片 | 列表显示“等待中”标签 | 总体状态=等待中；显示“任务已创建，等待执行” |
| `running` | 不在首页驻留展示 | 列表显示“运行中”标签 | 总体状态=运行中；显示轮询提示与最近刷新时间 |
| `success` | 不在首页驻留展示 | 列表显示“已完成”标签 | 总体状态=已完成；显示最终产物路径 |
| `failed` | 不在首页驻留展示 | 列表显示“失败”标签 + 失败阶段 | 总体状态=失败；显示失败原因 + 重跑入口 |
| `unknown` | 首页不渲染未知态 | 列表显示“未知状态”标签 | 总体状态=未知；提示刷新或稍后重试 |
| `empty` | 首页不渲染空态 | 列表显示空状态文案 + 返回首页创建入口 | 详情页不适用 |
| `network_error` | 表单区显示错误提示 | 列表区显示加载失败提示 | 详情区显示错误提示，不隐藏已加载信息 |
| `re_running` | 不在首页驻留展示 | 列表新增一条新任务记录 | 旧任务保留失败信息；新任务进入 `pending/running` |

## 3. 视觉设计系统

### 3.1 设计原则

- 低噪音：以清晰边界、温和层级和稳定留白承载复杂状态。
- 高对比：状态必须依赖“颜色 + 文本 + 图标/标签”共同表达，不能只靠颜色。
- 结构优先：卡片、列表、表单三类容器保持一致的视觉语言。
- 控制感：主按钮、禁用态、加载态、错误态需要明显差异。

### 3.2 Color Tokens

建议在全局样式中建立语义化 token，而不是直接依赖具体颜色值。

- 基础色
  - `--color-bg`: 页面背景
  - `--color-surface`: 卡片/面板背景
  - `--color-surface-elevated`: 浮层或强调卡片背景
  - `--color-border`: 默认边框
  - `--color-border-strong`: 强边框
  - `--color-text`: 主文本
  - `--color-text-muted`: 次级文本
  - `--color-text-subtle`: 弱提示文本

- 品牌与操作色
  - `--color-primary`: 主操作色
  - `--color-primary-hover`: 主操作 hover
  - `--color-primary-active`: 主操作按下
  - `--color-primary-contrast`: 主按钮文字

- 状态色
  - `--color-success`: 成功
  - `--color-warning`: 警告
  - `--color-danger`: 错误
  - `--color-info`: 信息

- 状态背景
  - `--color-success-bg`
  - `--color-warning-bg`
  - `--color-danger-bg`
  - `--color-info-bg`

- 状态使用规则
  - `pending`：使用中性色或信息色，强调“等待中”而不是“失败”
  - `running`：使用信息色或主色，强调“正在执行”
  - `success`：使用成功色
  - `failed`：使用错误色
  - 不允许只用颜色表达状态，必须同步显示文本标签

### 3.3 Spacing

采用 8px 基础间距体系，并允许 4px 作为微调单位。

- 建议 token
  - `--space-4`: 4px
  - `--space-8`: 8px
  - `--space-12`: 12px
  - `--space-16`: 16px
  - `--space-24`: 24px
  - `--space-32`: 32px
  - `--space-40`: 40px
  - `--space-48`: 48px

- 使用规则
  - 表单控件内边距统一，不在同类组件中出现“视觉跳步”
  - 卡片内内容间距优先使用 16 / 24
  - 页面模块之间优先使用 24 / 32

### 3.4 Radius

圆角以统一、克制为主。

- 建议 token
  - `--radius-sm`: 8px
  - `--radius-md`: 12px
  - `--radius-lg`: 16px
  - `--radius-pill`: 999px

- 使用规则
  - 输入框、按钮、标签保持同一圆角谱系
  - 卡片圆角应略大于控件圆角，形成层次
  - 不同圆角不应无意义混用

### 3.5 Typography

- 字体策略
  - 中文正文优先使用 `Noto Sans SC` 或同类无衬线中文字体
  - 英文与数字使用系统无衬线栈补足
  - 代码、路径、ID 使用等宽或近似等宽风格

- 字号层级
  - `H1`：页面主标题
  - `H2`：模块标题
  - `Body`：正文与说明
  - `Caption`：更新时间、辅助注释、轻提示

- 字重建议
  - 标题：600 / 700
  - 正文：400 / 500
  - 关键数字与状态：600

- 排版规则
  - 标题行高偏紧，说明文字行高偏松
  - 数字、路径、状态词优先保持完整可读，不做强制截断
  - 任务 ID、阶段名、路径名需要视觉上可快速识别

## 4. 交互规范

### 4.1 表单

- 表单默认单列布局，保证扫描效率。
- 每个字段都必须有显式标签，不依赖 placeholder 代替说明。
- 必填与可选要明确区分。
- 提交按钮在以下条件下置为禁用：
  - 必填项为空
  - 正在提交中
  - 前一次提交尚未完成

- 提交反馈
  - 点击后立即进入加载态
  - 按钮文案切换为“创建中…”或“重跑提交中…”
  - 表单不可重复提交
  - 成功后自动跳转，不要求用户手动刷新

### 4.2 加载

- 页面级加载
  - 详情页初次进入时应明确展示“加载中”状态
  - 加载过程不应造成布局跳动

- 局部加载
  - 表单提交时仅禁用当前操作区，不阻塞整页阅读
  - 重跑提交时，详情页其他信息保持可见

- 轮询加载
  - 运行中任务可使用轮询刷新阶段状态
  - 轮询状态必须对用户可见，避免“页面像卡住了”的误解

### 4.3 错误

- 错误必须可见、可读、可恢复。
- 错误容器应统一使用明确的错误样式，并配合文本说明。
- 错误文案应回答三个问题：
  1. 发生了什么
  2. 用户现在能做什么
  3. 是否可以重试

- 常见错误展示位置
  - 首页：表单下方
  - 详情页：总体状态区或重跑区上方
  - 接口失败：页面内可见，而不是仅写入控制台

### 4.4 重跑

- 只有任务失败时才展示重跑入口，避免干扰正常任务阅读。
- `start_stage` 需要使用下拉选择，且默认值优先指向失败阶段。
- 重跑表单必须告知：
  - 这是从某个阶段开始重跑
  - 不会修改原任务记录
  - 会生成新的任务实例

- 重跑确认原则
  - 按钮文案明确表达动作，例如“提交重跑”
  - 提交前不引入二次弹窗，避免打断高频故障处理
  - 如果未来增加高风险操作，再单独定义确认层级

## 5. 可访问性规范

### 5.1 动态反馈

- 所有错误提示区应使用 `role="alert"` 或 `aria-live="assertive"`。
- 轻提示、轮询更新、成功提示可使用 `aria-live="polite"`。
- 加载中状态应避免只有视觉变化，必要时补充可读文本。

### 5.2 焦点管理

- 表单提交后若出现错误，焦点应尽量回到错误区域或首个出错字段。
- 跳转到详情页后，页面主标题应成为首个可识别焦点锚点。
- 键盘用户必须能够完整完成创建、查看和重跑流程。

### 5.3 语义与控件

- 所有按钮都必须有明确可读标签，不能只靠图标。
- 所有输入框、下拉框都必须有对应的 `<label>`。
- 任务状态、失败原因、最终产物等信息应使用语义化元素组织。

### 5.4 对比度

- 正文与背景、按钮文字与按钮底色、状态标签与标签底色均需满足足够对比。
- 不能让浅色提示文案承载主要信息。
- 错误态、禁用态、低优先级说明都要保证在弱光或低质量显示器上仍可辨识。

## 6. 响应式与动效规范

### 6.1 响应式布局

- Mobile
  - 单列布局
  - 卡片与表单全宽
  - 按钮可点击区域足够大
  - 避免将关键信息压缩为两列小块

- Tablet
  - 保持以纵向阅读为主
  - 可在模块内部适度增加横向对齐，但不牺牲可扫读性

- Desktop
  - 允许更宽的内容区
  - 但正文行长应受控，避免过长造成阅读疲劳

- 断点原则
  - 先保证小屏完整可用，再增强大屏信息密度
  - 在小屏上优先保留状态、失败原因与操作入口

### 6.2 动效

- 动效目标是帮助理解状态变化，不是炫技。
- 推荐使用的动效
  - 按钮 hover / press 反馈
  - 卡片轻微浮起或边框变化
  - 新状态出现时的淡入

- 禁止项
  - 大幅位移动画
  - 影响阅读的长时延迟动画
  - 为轮询刷新引入显著闪烁

### 6.3 prefers-reduced-motion

- 必须支持 `@media (prefers-reduced-motion: reduce)`。
- 在 reduced motion 下：
  - 关闭或显著缩短过渡动画
  - 关闭无必要的位移动效
  - 保留必要的状态变化，但改为即时反馈

## 7. 文案与术语规范

### 7.1 统一词汇

以下词汇在全站统一使用，并与接口字段一一对应：

| 展示词 | 英文术语 | 后端字段 | 不推荐说法 |
|---|---|---|---|
| 任务 | Run | `run_id` | Job、执行实例 |
| 任务历史 | Run History | `/episodes/{episode_id}/pipeline-runs` | 历史记录列表（泛称） |
| 详情 | Run Detail | `/pipeline-runs/{run_id}` | 运行页面、日志页 |
| 重跑 | Retry From Stage | `start_stage` | 重试、重新执行（不带阶段） |
| 失败阶段 | Failed Stage | `failed_stage` | 报错步骤、异常节点 |
| 最终产物 | Final Outputs | `final_audio_path`/`final_video_path` | 输出文件（泛称） |
| Episode ID | Episode Identifier | `episode_id` | Episode、ID（孤立使用） |
| 状态 | Run State | `state` | 流程状态、任务进度（混用） |
| 更新时间 | Updated Time | `updated_at` | 最新时间、最近时间（无字段对应） |

### 7.2 推荐写法

- 推荐用：`创建任务`、`查看详情`、`提交重跑`、`任务历史`、`失败原因`
- 不推荐混用：`run`、`job`、`pipeline`、`执行实例`、`重试`、`重新执行`

### 7.3 文案原则

- 句子要短，优先一个动作一个句子。
- 错误文案要明确，不要只写“失败了”。
- 状态文案尽量使用用户能理解的词，而不是纯内部枚举。
- 路径、ID、阶段名保留英文或原始标识时，应配上中文解释。

## 8. 验收清单

### 8.1 首页

- [ ] 首屏在 `390x844` 视口下无需横向滚动即可看到主标题与创建表单首字段
- [ ] 表单中每个输入控件都存在可见 `<label>`，且提交按钮在必填项为空时 `disabled=true`
- [ ] 点击提交后 200ms 内按钮文案切换为“创建中...”，并阻止重复提交
- [ ] 错误提示容器存在 `role="alert"` 或 `aria-live="assertive"`
- [ ] 首屏存在“进入任务历史”入口，且链接包含 `episode_id` 参数
- [ ] 创建成功后 1s 内路由跳转到 `/runs/<run_id>?episode_id=<id>`

### 8.2 任务历史页

- [ ] 查询参数 `episode_id` 存在时，页面标题区显示当前 Episode 值
- [ ] 每条记录至少展示：`run_id`、状态标签、`updated_at`、详情入口
- [ ] `failed` 状态记录额外展示 `failed_stage` 文案
- [ ] 空列表时展示“先去首页创建任务”引导链接
- [ ] 在 `390px` 宽度下列表为单列，文本不重叠、按钮可点击

### 8.3 任务详情页

- [ ] 页面展示：总体状态、阶段状态列表、最终产物、失败原因（失败态）
- [ ] `running` 状态每 2s 轮询一次，并展示“加载中/更新中”可见提示
- [ ] 仅当 `state=failed` 时渲染重跑表单，其他状态不渲染
- [ ] 重跑表单默认值：优先 `failed_stage`，无值时退回 `tts_synthesis`
- [ ] 提交重跑后按钮切换“重跑提交中...”，成功后跳转新 `run_id` 详情页

### 8.4 视觉与可访问性

- [ ] 三页全部使用统一 token：`--color-*`、`--space-*`、`--radius-*`
- [ ] 状态标签同时包含文本（如“失败”）与颜色，不允许纯色块表达
- [ ] 错误区具备 `role="alert"` 或 `aria-live="assertive"`
- [ ] 键盘可完成“填写表单 -> 提交 -> 进入详情 -> 提交重跑”完整路径
- [ ] 正文对比度不低于 WCAG 4.5:1（正常文本）

### 8.5 响应式与动效

- [ ] `390px`（手机）、`768px`（平板）、`1280px`（桌面）三档均可完成主流程
- [ ] 小屏按钮高度不低于 `40px`，触达区域不低于 `40x40`
- [ ] 微动效时长 `120ms~240ms`，无大幅位移动画
- [ ] 存在 `@media (prefers-reduced-motion: reduce)` 并关闭非必要动画

### 8.6 文案

- [ ] “任务 / 任务历史 / 详情 / 重跑 / 失败阶段 / 最终产物”与术语表完全一致
- [ ] 页面不出现 `job`、`run`、`pipeline` 作为面向用户文案
- [ ] `Episode ID`、`start_stage` 首次出现时附带中文说明

## 9. 风险与边界

- 编码风险：所有前端文本文件统一 UTF-8，无 BOM；新增 smoke 检查中文关键文案。
- 长文本风险：路径或错误消息过长时需要自动换行，禁止遮挡按钮。
- 时区风险：时间展示统一使用 `zh-CN` 本地化格式，避免混用 UTC 原始串。
- A11y 风险：所有错误提示必须可被读屏识别，避免仅红色边框提示。
- 小屏风险：不得在小屏将关键状态与操作拆成多列导致点击困难。
